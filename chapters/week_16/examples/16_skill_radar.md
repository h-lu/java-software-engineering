# 示例 3：能力雷达图与成长轨迹

## 问题场景

Week 16 结束时，老师让你写"课程总结"。你列了 10 条"学到了什么"，但简历上怎么写？

"会 Java"？太泛了。
"会写 REST API"？太基础了。

**能力雷达图**（Skill Radar）是一种可视化方法——把你的能力按维度分类，每个维度打分（1-5 分），连成一个多边形。

它的价值：
1. **自我认知**：看清自己的强项和弱项
2. **成长轨迹**：Week 01 vs Week 16 对比，看见进步
3. **简历素材**：雷达图 + 具体证据，比"精通 Java"有说服力

## 核心概念

**六维能力模型**：针对软件工程入门课程，建议评估 6 个维度：

| 维度 | 1 分（入门） | 3 分（熟练） | 5 分（精通） |
|------|-------------|-------------|-------------|
| **编程语言** | 能写简单的类和方法 | 理解泛型、异常、集合 | 能设计库级别的 API |
| **工程工具** | 会用 Git 基本命令 | 熟悉分支工作流、PR 流程 | 能配置 CI/CD |
| **测试能力** | 会写简单的断言 | 能写参数化测试、Mock | 能设计测试策略 |
| **架构设计** | 能实现需求 | 能用设计模式重构 | 能做技术选型 |
| **文档能力** | 能写 README | 能写 API 文档、ADR | 能写技术博客、教程 |
| **团队协作** | 能独立完成任务 | 能参与 Code Review | 能带领小型团队 |

- 团队协作：1 分（习惯单打独斗）

**短板**：完全不懂工程化，以为"写代码 = 全部"

- 文档能力：3 分（能写 API 文档、ADR，但博客还需要练习）
- 团队协作：3 分（能做 Code Review，能当首席架构师）

**成长**：从"只会写代码"到"会做工程"

## 各维度详细评估

### 编程语言：2 → 4

**Week 01**：
- 只会 `if/else` 和 `for` 循环
- 不知道什么叫"受检异常"
- 看到泛型 `<String>` 觉得"这什么语法"

**Week 16**：
- 熟练使用 ArrayList、HashMap、Stream API
- 能设计 `BookingService` 这样的服务层 API
- 知道什么时候用 `Optional` 避免空指针
- 能看懂 `@FunctionalInterface` 和 Lambda 表达式

**证据**：
- CampusFlow 的 `BookingService` 有 15 个公开方法，API 设计清晰
- Week 08 用策略模式重构了"预约规则"，代码可扩展

**下一步（5 分）**：
- 学习 Java 8 新特性（CompletableFuture、Optional）
- 研究 Spring Boot 的依赖注入原理

---

### 工程工具：2 → 4

**Week 01**：
- 只会 Git 基本命令，不知道 branch 是什么
- 从没听过 Maven
- 写代码用记事本 + 命令行 `javac`

**Week 16**：
- 熟练 Feature Branch 工作流，能解决合并冲突
- 能写 `pom.xml`，添加依赖、配置插件
- 用 Maven 打包可执行 JAR，一键部署
- 本地质量门禁：`mvn test` + `mvn spotbugs:check` 通过才提交

**证据**：
- Git 历史清晰，每个功能一个分支
- Week 14 用 Maven Shade Plugin 打包，用户只需 `java -jar`
- Week 11 配置 JaCoCo，覆盖率 < 80% 自动失败

**下一步（5 分）**：
- 学习 GitHub Actions，配置自动化 CI/CD
- 尝试 Docker 容器化部署

---

### 测试能力：1 → 4

**Week 01**：
- 不知道什么叫"单元测试"
- 只会用 `System.out.println` 调试

**Week 16**：
- 能写 JUnit 5 测试：`@Test`、`@BeforeEach`、`@ParameterizedTest`
- 熟悉断言：`assertEquals`、`assertThrows`、`assertTrue`
- 能用 Mockito Mock 依赖
- 测试覆盖率 87%，测试用例 127 个

**证据**：
- `BookingServiceTest` 有 23 个测试用例，覆盖所有分支
- Week 12 Bug Bash 时，有测试的模块 0 bug
- 用 `@ParameterizedTest` 测试 8 种预约冲突场景

**下一步（5 分）：
- 学习 Testcontainers，做真数据库集成测试
- 研究测试金字塔：单元测试 vs 集成测试 vs E2E 测试的比例

---

### 架构设计：1 → 3

**Week 01**：
- 不会"设计"，只会"写代码"
- 所有代码塞在 `main()` 方法里

**Week 16**：
- 能用"名词提取法"做领域建模
- 写了 4 篇 ADR，记录架构决策
- 能用策略模式、Repository 模式
- 知道 SOLID 原则（虽然偶尔还是会违反）

**证据**：
- ADR-001 设计了 User / Room / Booking 三层模型
- Week 08 用策略模式重构"预约规则"，`BookingValidator` 接口有 3 个实现
- CampusFlow 分层：Controller → Service → Repository

**不足**：
- 有时候为了"快点做完"，会违反单一职责原则
- `Booking` 类有点臃肿（15 个方法），可以考虑拆分

**下一步（4-5 分）**：
- 学习 DDD（领域驱动设计）
- 研究微服务架构 vs 单体应用的取舍

---

### 文档能力：1 → 3

**Week 01**：
- 从没写过 README
- 注释只在老师要求时才写

**Week 16**：
- 能写结构清晰的 README：快速开始、功能特性、API 文档
- 能写 ADR（架构决策记录），4 篇共 3000 字
- 能用 OpenAPI 规范写 API 文档
- 注释覆盖率 > 60%

**证据**：
- README 包含：快速开始、功能列表、API 文档、常见问题
- ADR-003 的 API 设计决策被评委表扬"逻辑清晰"
- Week 13 用 Swagger UI 自动生成交互式文档

**不足**：
- 注释有时候是"翻译代码"（如 `// 设置用户名`），应该解释"为什么"
- 没写过技术博客，不知道如何向大众讲解技术

**下一步（4-5 分）**：
- 学习"文档审查检查清单"
- 写一篇技术博客，讲"CampusFlow 的架构演进"

---

### 团队协作：1 → 3

**Week 01**：
- 习惯一个人写代码
- 不知道 Code Review 是什么

**Week 16**：
- 能参与 Code Review，给出建设性反馈
- 当过 2 轮首席架构师（Week 02-03, Week 08-09）
- 能解决 Git 合并冲突
- 能做技术演示（8 分钟讲完 CampusFlow）

**证据**：
- Week 04 的 PR 被老潘表扬"变量命名清晰"
- Week 06 当首席架构师时，主持了"领域模型重构"讨论
- Week 15 演示时，和队友配合，Q&A 互相补充

**不足**：
- 有时候还是会忍不住自己写代码，而不是让队友尝试
- Q&A 环节被问"压力测试怎么做"，答不上来

**下一步（4-5 分）**：
- 学习如何带领 3-5 人的小团队
- 练习"如何拒绝不合理的需求"

## 简历上的写法

**❌ 不要写**：
> "精通 Java，会写 REST API，有团队协作经验"

**✅ 建议写**：
> "16 周团队项目 CampusFlow（会议室预约系统）：
> - 技术栈：Java 17 + Javalin + SQLite + JUnit 5
> - 测试：编写 127 个测试用例，覆盖率 87%，熟悉 Mock 和参数化测试
> - 工程实践：Feature Branch 工作流、PR Code Review、ADR 架构决策记录
> - 架构设计：用策略模式重构预约规则，API 遵循 RESTful 风格
> - 工具：Maven 打包部署、本地质量门禁（JaCoCo + SpotBugs）
> - 团队协作：担任 2 轮首席架构师，主导领域模型与架构演进"
