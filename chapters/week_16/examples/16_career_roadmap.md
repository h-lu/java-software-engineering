# 示例 5：工程师职业发展路径

## 问题场景

Week 16 结束时，小北问老师："接下来我应该学什么？"

老师反问："你想成为什么样的工程师？"

小北愣住了——他只知道自己"会写 Java"，但不知道工程师的职业路径有哪些选择。

**职业发展路径**（Career Roadmap）不是"5 年后成为技术总监"，而是清晰地看见"从现在到下一级，我需要补什么能力"。

## 核心概念

**工程师成长的三个阶段**：

| 阶段 | 关注点 | 典型问题 | 能力要求 |
|------|--------|----------|----------|
| **初级工程师** | "怎么实现？" | 这段代码怎么写？ | 编程语言、基本工具 |
| **中级工程师** | "怎么做好？" | 这个设计合理吗？ | 架构设计、工程实践 |
| **高级工程师** | "怎么系统性地解决？" | 如何避免类似问题再次出现？ | 系统设计、技术领导力 |

## 使用示例

下面是小北（Week 16 水平）到高级工程师的成长路径：

```markdown
# 工程师职业发展路径

## 当前水平：接近中级工程师

**Week 16 的能力雷达图**：
- 编程语言：4 分
- 工程工具：4 分
- 测试能力：4 分
- 架构设计：3 分
- 文档能力：3 分
- 团队协作：3 分

**平均分：3.5/5** → 对应职位：**中级工程师（Junior-Mid 过渡期）**

---

## 路径一：中级工程师（目标：3-6 个月达到）

### 能力要求

| 维度 | 目标分数 | 具体标准 |
|------|----------|----------|
| 编程语言 | 4 分 | 熟悉 Java 核心 API，能做技术选型 |
| 工程工具 | 4 分 | 熟练 Git、Maven、本地质量门禁 |
| 测试能力 | 4 分 | 能写覆盖 80%+ 的测试，会用 Mock |
| 架构设计 | 3 分 | 能用设计模式重构，能写 ADR |
| 文档能力 | 3 分 | 能写 API 文档、ADR |
| 团队协作 | 3 分 | 能参与 Code Review，能当首席架构师 |

### 差距分析

**小北的短板**（3 分 → 需要提升）：
- 架构设计：有时候还是会为了"快点做完"违反 SOLID 原则
- 文档能力：注释有时候是"翻译代码"，没解释"为什么"
- 团队协作：Q&A 环节被问住，说明知识体系有盲区

### 3 个月行动计划

**Month 1：巩固基础**
- [ ] 重读《代码整洁之道》，每周重构 1 个 CampusFlow 类
- [ ] 学习 Java 8 新特性（Optional、Stream、CompletableFuture）
- [ ] 练习 TDD：选一个小功能（如"预约提醒"），先写测试再写代码

**Month 2：扩展工具链**
- [ ] 学习 Docker：把 CampusFlow 容器化部署
- [ ] 配置 GitHub Actions：自动运行测试 + 覆盖率检查
- [ ] 尝试 Postman：手动测试 API 性能（对比 JUnit 集成测试）

**Month 3：深入架构**
- [ ] 学习 Spring Boot：对比 Javalin，理解 IoC 和 DI
- [ ] 研究 MySQL：把 SQLite 数据迁移到 MySQL，对比差异
- [ ] 设计一个"扩展功能"（如预约提醒），写完整的 ADR

**验证标准**：
- 能独立负责一个模块（如"用户认证"）的技术选型
- 能在 Code Review 中给出"设计层面"的建议，而不只是"命名建议"
- 能向新人讲解 CampusFlow 的架构设计

---

## 路径二：高级工程师（目标：1-2 年达到）

### 能力要求

| 维度 | 目标分数 | 具体标准 |
|------|----------|----------|
| 编程语言 | 4 分 | 精通一门语言，了解多门语言 |
| 工程工具 | 4-5 分 | 能配置 CI/CD，能选择合适的工具 |
| 测试能力 | 4 分 | 能设计测试策略（单元/集成/E2E 的比例） |
| 架构设计 | 3-4 分 | 能做技术选型，能设计系统架构 |
| 文档能力 | 3-4 分 | 能写技术博客，能做技术分享 |
| 团队协作 | 3-4 分 | 能带领 3-5 人小团队 |

### 差距分析

**从中级到高级，核心变化**：
- 不只是"实现功能"，而是"设计系统"
- 不只是"写代码"，而是"影响团队"
- 不只是"解决问题"，而是"预防问题"

### 6-12 个月行动计划

**Phase 1：系统设计（3 个月）**
- [ ] 学习系统设计：《数据密集型应用系统设计》
- [ ] 设计一个"分布式 CampusFlow"：
  - 用 Redis 缓存热门会议室
  - 用消息队列处理"预约提醒"
  - 用 Nginx 做负载均衡
- [ ] 写一篇技术博客："从单体到分布式：CampusFlow 架构演进"

**Phase 2：技术领导力（3 个月）**
- [ ] 参加 2 个开源项目（从修文档开始）
- [ ] 在公司内做一次技术分享（如"如何做 Code Review"）
- [ ] 带一个新人（让 TA 从 Week 01 开始学 CampusFlow）
- [ ] 学习如何"拒绝不合理的需求"（Week 16 反馈里有）

**Phase 3：深度工程实践（3 个月）**
- [ ] 配置完整的 CI/CD：
  - 自动运行测试
  - 自动部署到测试环境
  - 性能测试（JMeter）
- [ ] 性能优化：
  - 用 JaCoCo 找慢查询
  - 用 Profiler 找 CPU 瓶颈
  - 从 API 2 秒响应优化到 200 ms
- [ ] 安全加固：
  - 加入 Spring Security（如果用 Spring Boot）
  - 做 XSS/CSRF 防护
  - 写一份"安全检查清单"

**Phase 4：扩展视野（3 个月）**
- [ ] 学习第二门语言（Python 或 Go）
- [ ] 研究微服务架构（对比单体应用）
- [ ] 阅读《凤凰项目》（理解 DevOps 文化）
- [ ] 参加 1 个技术会议（如 QCon、ArchSummit）

**验证标准**：
- 能独立设计一个中型系统（1000+ 用户，10+ API）
- 能主导技术选型（如"为什么用 Kafka 而不是 RabbitMQ"）
- 能带领 3-5 人团队完成一个项目

## 路径三：技术专家 / Tech Lead（目标：2-5 年达到）

### 能力要求

| 维度 | 目标分数 | 具体标准 |
|------|----------|----------|
| 编程语言 | 5 分 | 能设计库级别的 API，能写框架 |
| 工程工具 | 4-5 分 | 能评估和引入新工具，能改进工作流 |
| 测试能力 | 4-5 分 | 能定义团队测试规范，能设计测试框架 |
| 架构设计 | 4-5 分 | 能设计大型系统架构，能做技术战略 |
| 文档能力 | 4-5 分 | 能写 RFC、技术白皮书，能影响业界 |
| 团队协作 | 4-5 分 | 能带领 10+ 人团队，能培养下属 |

### 差距分析

**从高级到专家，核心变化**：
- 不只是"设计系统"，而是"定义技术方向"
- 不只是"带领团队"，而是"培养团队"
- 不只是"解决当下问题"，而是"预见未来问题"

### 2-5 年行动计划

**Year 1：深度聚焦**
- [ ] 选择一个领域深耕（如"分布式系统"或"数据库"）
- [ ] 在该领域达到"专家"水平：
  - 如果是分布式系统：读 CAP 论文、研究 Raft 协议、实践 etcd
  - 如果是数据库：研究 B+ 树、MVCC、事务隔离级别
- [ ] 发表技术影响力：
  - 写 10 篇高质量技术博客
  - 在 GitHub 上有 500+ stars 的项目
  - 在技术会议做分享

**Year 2：技术战略**
- [ ] 参与/主导公司技术战略（如"从单体迁移到微服务"）
- [ ] 定义团队工程标准（如"所有 API 必须有 OpenAPI 文档"）
- [ ] 培养下一代工程师：
  - 做技术导师（带 2-3 个新人）
  - 设计内部培训课程
  - 建立技术晋升标准

**Year 3+：行业影响力**
- [ ] 在开源社区做 Maintainer
- [ ] 参与技术标准制定（如 OpenAPI 规范）
- [ ] 写书或在顶级会议（如 VLDB、OSDI）发论文

## 路径四：转型方向（非纯技术）

如果你发现"写代码"不是唯一的热情，还有这些选择：

### 方向一：产品经理（PM）
**能力迁移**：
- 架构设计 → 产品设计
- 用户反馈 → 需求分析
- ADR → PRD（产品需求文档）

**适合人群**：
- 喜欢和用户聊、理解业务痛点
- 对"做什么"比"怎么做"更感兴趣
- Week 16 项目集市上，Q&A 环节表现好

**第一步**：
- 转 PM 前先做 1 年"技术型 PM"（仍然写代码，但参与产品讨论）
- 学习《启示录》《人人都是产品经理》

---

### 方向二：技术销售 / 解决方案架构师
**能力迁移**：
- 技术演示 → 客户演示
- 架构设计 → 解决方案设计
- Q&A 应对 → 客户答疑

**适合人群**：
- 喜欢和人沟通、演讲能力强
- Week 15 演示时自信、自然
- 能把复杂技术讲得简单

**第一步**：
- 参与售前演示（配合销售团队）
- 学习如何做 PoC（概念验证）

---

### 方向三：技术创业
**能力迁移**：
- CampusFlow 完整项目 → MVP（最小可行产品）
- 项目集市反馈 → 用户调研
- 团队协作 → 创业团队

**适合人群**：
- 有强烈的问题解决欲望
- 不怕风险、抗压能力强
- Week 16 反馈里有"这个应该做成产品"的评价

**第一步**：
- 把 CampusFlow 开源，积累用户
- 学习《精益创业》《从 0 到 1》

## 证书与学历

**有用但不是必须**：
- **证书**：
  - Oracle Certified Professional（Java 开发者）
  - AWS Certified Solutions Architect（云架构师）
- **学历**：
  - 硕士（计算机相关）—— 算法、系统设计更扎实
  - MBA —— 如果想转型技术管理

**老潘的建议**：
> "证书是加分项，不是门票。我见过有证书但写不出好代码的人，也见过无证书但架构设计一流的人。重点是'你能做什么'，不是'你有什么证书'。"

## 薪资参考（2026 年国内市场）

| 职位 | 经验 | 月薪范围（税前） |
|------|------|------------------|
| 初级工程师 | 0-1 年 | 8k - 15k |
| 中级工程师 | 1-3 年 | 15k - 25k |
| 高级工程师 | 3-5 年 | 25k - 40k |
| 技术专家 | 5-8 年 | 40k - 70k |
| Tech Lead | 5-10 年 | 50k - 100k |

**影响薪资的因素**：
- 城市（北京/上海/深圳 vs 二线城市）
- 公司（大厂 vs 创业公司）
- 学历（硕士溢价约 10-20%）
- 业务方向（AI/区块链薪资溢价）

## 推荐资源

### 书籍
**必读**：
- 《代码整洁之道》—— Week 02 的 SOLID 原则深入
- 《重构》—— Week 08 的重构方法论
- 《设计模式》—— 策略模式、工厂模式的经典讲解

**视野**：
- 《程序员修炼之道》—— 工匠精神
- 《五十年后》—— 软件工程的本质思考

- **技术博客**：美团技术博客、阿里云栖社区、Uber Engineering Blog
- **开源项目**：GitHub Trending，每周找一个感兴趣的项目

- **LeetCode**：保持算法敏感度（每周 2-3 题）

## 最终建议

**小北的 3 年规划**（从 Week 16 出发）：

**Year 1：巩固基础 + 扩展工具**
- 重构 CampusFlow 到"自己满意"的状态
- 学习 Spring Boot + Docker + MySQL
- 写 5 篇技术博客

**Year 2：系统设计 + 技术领导力**
- 设计一个"分布式 CampusFlow"
- 带一个 3 人小团队（可能是 CampusFlow 2.0）
- 参与开源项目

**Year 3：深度聚焦 + 行业影响力**
- 选择一个领域深耕（如"分布式系统"）
- 在技术会议做分享
- 发起一个开源项目

**3 年后的目标**：高级工程师（平均 4.5/5 分）
