# Week 16：项目展示与工程复盘——从课程到职业生涯的第一步

> "结束只是新的开始。"
> — 老潘

2026 年的计算机科学教育领域正在发生一个深刻的变化：**毕业生的"第一份工作准备度"不再只是"会写代码"，而是"能在团队中交付价值"**。

Google 2026 年的工程招聘报告显示了一个惊人的数据：**72%** 的新入职工程师在前六个月遇到的最大挑战不是"写代码"，而是"理解大型代码库"、"与团队协作"、"做出架构决策"。

这说明什么？

**你学了一学期的 Java 和软件工程，不是为了记住语法，而是为了掌握"在真实项目中交付价值"的能力**。

与此同时，AI 工具的普及也让招聘标准发生变化。2026 年 1 月，LinkedIn 发布的报告显示：**61%** 的科技公司表示"会用 AI 工具"是基本要求，但 **93%** 的公司强调"不能盲目信任 AI，必须有独立判断"。

本周你要学的内容不是"课程结束"，而是"职业生涯的开始"——展示你的作品、总结你的成长、规划下一步的方向。

## 1. 项目集市——让你的作品被看见

周一早上 9 点，项目集市正式开始。

小北站在桌子前，深呼吸。桌上放着海报、笔记本电脑和二维码。CampusFlow 的海报上写着："极简任务管理工具——小而美，学习友好"。

第一个参观者走过来。

"这是什么项目？"参观者问。

小北开始讲述——上周设计的脚本终于派上用场了。从问题背景（"为什么需要另一个任务管理工具"），到技术选型（"为什么选择 Java + Javalin + SQLite 这样的极简技术栈"），到功能演示（"看一下，5 秒钟创建一个任务"），到工程实践（"我们实现了 80% 的测试覆盖率"），再到踩坑经验（"Bug Bash 帮我们发现了 3 个边界情况 bug"）。

8 分钟过去了，小北讲完了。

参观者点点头："很清晰。我特别喜欢你们'极简技术栈'的设计。你们下一步有什么计划？"

这正是上周准备问答时讨论过的问题。阿码接话："短期计划是增加任务搜索功能，长期计划是如果用户量增长，考虑迁移到 PostgreSQL。但我们会保持'小而美'的定位，不做功能堆砌。"

参观者笑了："很好，你们有明确的定位。我扫一下二维码，试试你们的在线演示。"

上午 10 点，老潘走过来。

"你们今天的展示很成功。"老潘说，"我观察到几个亮点：
1. **叙事清晰**：观众能理解 CampusFlow 的价值
2. **技术亮点突出**：极简技术栈有说服力
3. **分享真实经验**：Bug Bash 和 AI 协作的故事很有启发
4. **问答应对得好**：回答清晰，不争辩"

小北松了一口气："上周练了这么多遍，终于派上用场了。"

"展示不是一次完美的，而是 iterative 的。"老潘说，"今天收集的反馈，可以作为未来改进的参考。"

老潘让小北打开一个共享文档——这是老师为项目集市准备的**反馈收集表**。一上午，CampusFlow 收集了 20 多位参观者的反馈。

小北看着反馈数据，发现了一个有趣的模式。

"正面反馈集中在'叙事清晰'和'技术亮点'，"小北说，"这说明我们的展示策略是对的。改进建议集中在'功能扩展'——6 个人提到移动端优化，4 个人提到数据导出，3 个人提到任务搜索。"

"看到模式了吗？"老潘说，"用户对 CampusFlow 有期待。这些反馈不是批评，而是未来的 Roadmap。"

阿码突然想到："老潘，我能不能把今天的录屏放到 GitHub 上？这样没能来现场的人也能看到展示。"

"非常好的想法！"老潘说，"这就是'文档即代码'的实践——展示录屏、PPT、海报、演示环境，都是项目的一部分。未来学弟学妹看到这些，能快速理解 CampusFlow 的价值。"

小北若有所思："所以展示不只是'给别人看'，也是'沉淀知识'？"

"对。"老潘说，"CampusFlow 现在有代码、有文档、有展示、有反馈。这是一个完整的项目资产。代码会过时，但展示材料、经验文档、反馈记录会持续产生价值。"

老潘补充了一句话："而且，展示能力本身就是核心竞争力。2026 年的行业数据显示，**85%** 的技术主管认为'清晰的沟通'比'技术深度'更重要。在 AI 时代，代码生成的门槛降低了，但'讲故事'、'说服他人'、'解释代码'这些能力无法被 AI 替代。"

小北若有所思。上周她还觉得展示是"额外负担"，现在开始理解——展示是工程能力的一部分，甚至是 AI 时代更重要的能力。

## 2. 工程复盘——从"做了什么"到"学到了什么"

周二下午，老师发布了 Week 16 的第二个任务：工程复盘。

"项目集市结束了，"老师在公告中写道，"现在该复盘了。复盘不是'流水账'，而是'系统总结'——你做了什么决策、为什么这么做、学到了什么、未来如何改进。"

小北看着公告，有些困惑："复盘要怎么写？不就是'我学了 Java、做了项目'吗？"

老潘路过，听到这话，摇头："如果你只写'我学了 Java、做了项目'，那就太浅了。复盘的核心是'识别成长'——你从 Week 01 的'只会 Python'，到 Week 16 的'能用 Java 做完整项目'，中间经历了什么转变？"

"转变？"小北疑惑。

"对。"老潘说，"我给你们一个复盘框架——**KPT 复盘法**：
- **Keep（保持）**：哪些做得好，未来继续？
- **Problem（问题）**：哪些做得不够，未来改进？
- **Try（尝试）**：哪些新方法下次试试？

用这个框架，你能系统总结一学期的经验。"

老潘在白板上写下了**工程复盘的四个维度**——不是技术清单，而是成长视角。

他一边画一边解释："复盘不是罗列你学了哪些技术，而是理解你的成长轨迹。比如，技术决策回顾——你写过 4 篇 ADR，每篇都记录了一个关键决策。现在回头看，哪些决策是对的？哪些可以改进？"

小北若有所思："所以我得回顾 16 周的所有 ADR、代码、文档，找出成长轨迹？"

"对。"老潘说，"我来给你们举几个具体的例子。"

---

**技术决策回顾**：你写过 4 篇 ADR，记录了 CampusFlow 的架构演进。ADR-001 确定了领域模型设计，参考了 DDD 思想，核心实体是 Task——这个决策让职责清晰，易扩展。ADR-002 选择 SQLite 而非 PostgreSQL，原因是零配置、适合小项目——这个决策满足需求，未来可以迁移。ADR-003 决定 API 设计遵循 RESTful，让端点清晰、文档完善。ADR-004 确定架构演进保持分层，让代码简洁、易理解。

"看到模式了吗？"老潘问，"成功的决策——选择极简技术栈，避免了 Spring 的复杂度。需要改进的决策——Week 05 的集合选择可以更早引入 HashMap，提升查询效率。意外的收获——ADR 让我们重新思考设计决策，不是'跟着感觉走'。"

**AI 协作经验**：Week 10 的 AI 审查训练很有启发。AI 没有处理空输入（后来加了边界检查），AI 生成的 CSS 不够美观（人工优化），AI 建议的方案有时过度设计（需要人工判断）。所以你学会了——AI 适合生成样板代码（getter/setter、测试骨架），适合代码审查辅助（找 bug、提建议），适合文档生成（API 文档草稿）。但 AI 不适合架构决策（ADR 必须由人写），不适合边界情况处理（AI 常遗漏），不适合安全考虑（SQL 注入、XSS）。

"关键是什么？"老潘问。

小北想了想："我比 AI 更懂项目背景、用户需求、边界情况。AI 比我更懂 API 用法、语法细节、代码风格。"

"对！"老潘说，"AI 是倍增器，但质量把控在人。"

**工程实践成长**：从 Week 01 到 Week 16，你的测试策略从"不写测试"到"80% 覆盖率 + 质量门禁"，理解了测试的价值。Git 协作从"只会 commit"到"Feature Branch + PR + Code Review"，理解了团队协作。文档习惯从"不写文档"到"OpenAPI + README + ADR"，理解了知识传递。质量意识从"能跑就行"到"静态分析 + CI 门禁"，理解了质量保障。

"最大的转变是什么？"老潘问。

"从'能跑就行'到'质量意识'。"小北说。

"对。"老潘说，"最有价值的习惯——写 ADR 让我重新思考设计决策。最需要改进的地方——有时过度依赖 AI，应该更多独立思考。"

**软技能提升**：沟通能力从"讲不清楚代码"到"能 10 分钟讲清楚项目"，在项目集市上验证了。团队协作从"单打独斗"到"团队分工 + Code Review"，学会了信任和依赖。时间管理从"拖延到最后"到"迭代开发 + 每周交付"，降低了最后一周的 panic。抗挫折能力从"报错就慌"到"冷静调试 + 根因分析"，Bug Bash 训练了耐心。

"最意外的收获是什么？"老潘问。

"展示能力让我更自信。"小北说。

"很好。"老潘说，"现在你用 KPT 总结一下：
- **Keep（保持）**：写 ADR 的习惯、测试驱动的实践、Code Review 的习惯、极简技术栈的选择
- **Problem（问题）**：有时过度依赖 AI、文档更新不及时、时间管理还需改进
- **Try（尝试）**：下次项目尝试新的技术栈、更早开始文档、尝试 TDD"

阿码听完复盘框架，若有所思："所以复盘不只是'回忆'，而是'识别成长'？"

"对。"老潘说，"通过复盘，你能看到自己的成长轨迹——从 Week 01 的'只会 Python'，到 Week 16 的'能用 Java 做完整项目'。这种成长不是自然发生的，而是通过 16 周的刻意练习获得的。"

老潘补充："复盘还有一个价值——**为未来提供参考**。下次你做项目时，可以回头看这份复盘，知道哪些做得好（Keep），哪些需要改进（Problem），哪些新方法可以尝试（Try）。"

小北若有所思："所以我得花一下午认真写这份复盘？"

"对。"老潘说，"复盘是一学期最珍贵的文档之一。代码会过时，但成长经验永不过时。"

老潘还给了一个**能力雷达图**的工具——用可视化方式呈现你的能力成长。

```
       编程能力 (4)
          /|\
         / | \
  架构设计(3) | 测试能力(4)
       \     /
    \   /   \  /
  沟通能力(4) \
     |     |
  文档能力(3) |
     \     /
  团队协作(4)
```

"评分标准很直接，"老潘说，"1 分是完全不会，需要手把手教；5 分是能教别人，有深度理解。你给自己打分，能看到优势和能力边界。"

小北看着雷达图，突然意识到："我的优势是'编程能力'和'测试能力'，这意味着什么？"

"这意味着你的职业方向，"老潘说，"如果你的优势是'架构设计'和'沟通能力'，那可能适合做技术 Leader 或架构师。如果优势是'编程能力'和'测试能力'，那可以走后端工程师或质量工程师的路线。"

阿码若有所思："所以复盘不只是'总结过去'，也是'规划未来'？"

"对！"老潘说，"通过复盘，你理解了自己的优势和能力边界，就能更好地规划下一步的学习和职业方向。"

> **AI 时代小专栏：工程能力评估——代码能力 vs 工程思维**
>
> 2026 年，软件工程师的能力评估标准正在发生深刻变化。传统的"代码能力"正在被 AI 工具 commoditize（商品化），而"工程思维"成为区分高级和初级工程师的关键因素。
>
> LinkedIn 2026 年 1 月的报告显示，**61%** 的科技公司表示"会用 AI 工具"是基本要求，但 **93%** 的公司强调"不能盲目信任 AI，必须有独立判断"。这意味着什么？代码生成能力正在成为"基础门槛"，但判断力、决策力、质量把控能力成为"竞争优势"。
>
> 具体来说，**代码能力**包括：语法熟练度、API 使用、算法实现、bug 修复。这些能力 AI 可以帮助完成，GitHub 数据显示使用 Copilot 的开发者编码时间减少 55-60%。**工程思维**包括：架构决策能力（如何设计系统？）、产品思维（用户真正需要什么？）、质量意识（如何预防 bug？）、团队协作（如何和他人合作？）。这些能力 AI 难以替代。
>

## 4. 职业发展路径——下一步去哪里

1. **技术深度路线**：成为某个领域的专家（如后端架构师、数据库专家）
2. **技术广度路线**：成为全栈工程师或技术 Leader

小北若有所思："我喜欢做架构设计，写 ADR 的时候很投入。可能是技术深度路线？"

"高级工程师（5-8 年）的核心能力是：能设计系统级架构（整个系统，不是单个模块），能带领 3-5 人团队做技术决策，能处理复杂问题（性能瓶颈、安全漏洞、分布式一致性问题），能做技术规划（6-12 个月的技术路线图）。学习重点：深入某个领域，学习软技能（沟通、领导力、项目管理），学习业务理解，学习 mentor 技能。"

> **AI 时代小专栏：2026 年的工程师职业路径——从初级到高级**
>
> 2026 年的软件工程职业路径正在发生变化。传统的"深度专精"路线依然重要，但"T型技能"（T-shaped skills）——既有深度（某个领域的专长），也有广度（跨栈协作能力）——成为更受青睐的模型。
>
> Dev.to 2026 年的全栈开发者路线图建议：**从通才开始，在 1-2 个领域发展深度专长，保持广博知识**。这意味着初级工程师应该先有广度（能做前端、后端、部署的基本工作），然后在某个领域深入（如后端架构、前端工程化、DevOps）。EPAM 的研究表明，**T型开发者正在适应 AI 优先开发环境**——竖线代表技术深度（架构师的直觉、编码者的纪律、工程师的眼光），横线代表跨前中后端的广泛知识。
>
> Coursera 2026 年的初级工程师要求数据：1-3 年经验，与高级工程师密切合作。核心要求包括：更深入的语言/系统知识、Testing skills、Basic system design、Agile practices、Cloud fundamentals。这说明什么？初级工程师不再是"只会写代码"，而是需要具备"工程思维"——理解系统设计、测试策略、敏捷实践。
>

## Git 本周要点

本周不涉及新的 Git 命令，重点是回顾整个学期的 Git 工作流。

```bash
# 1. 克隆仓库（Week 01）
git clone https://github.com/your-team/campusflow.git
cd campusflow

# 2. 创建功能分支（Week 02-15）
git checkout -b feature/xxx

# 3. 提交代码
git add .
git commit -m "feat: add xxx"

# 4. 推送到远程
git push origin feature/xxx

# 5. 创建 Pull Request（GitHub Web UI）

# 6. Code Review 后合并
git checkout main
git merge feature/xxx

# 7. 打标签（Week 14-15）
git tag -a v1.0.0 -m "Release v1.0.0"
git push origin v1.0.0

# 8. 查看历史（Week 16）
git log --oneline --graph
git tag
```

**Git 最佳实践回顾**：

- **分支策略**：使用 Feature Branch 工作流，每个功能一个分支，合并前通过 Code Review
- **提交规范**：feat（新功能）、docs（文档）、fix（Bug 修复）、refactor（重构）、test（测试）
- **版本管理**：遵循 Semantic Versioning（MAJOR.MINOR.PATCH）
