## 【主体】7 分钟

### 1. 问题背景（1 分钟）

> Week 01 选题时，我们发现市面上很多任务管理工具（如 Todoist、Notion）功能很强大，但代码很复杂——Todoist 的前端代码超过 10 万行，Notion 更是百万级行数。新人上手需要几个月，甚至看不懂整体架构。
>
> 我们想：**能不能做一个"小而美"的任务管理工具，既能用，又能作为学习案例？**
>
> 于是 CampusFlow 诞生了：核心功能完整（创建、编辑、标记完成、删除），但代码不超过 2000 行，架构清晰、文档完善，适合学习和扩展。

---

### 2. 技术亮点（2 分钟）

#### 技术选型（1 分钟）

> 我们选择了**极简技术栈**——Java + Javalin + SQLite。
>
> **为什么用 Java？** 我们团队有 Java 基础，而且 Java 的静态类型系统更适合学习工程化开发。Python 快速原型好，但大型项目维护成本高；Go 性能好，但生态不如 Java。对于学习项目，Java 是平衡选择。
>
> **为什么不用 Spring？** Spring 是企业级框架，功能强大但学习曲线陡峭。CampusFlow 的目标是"代码简洁易懂"，Javalin 只有几百 KB，但足够 Web 开发。我们更关注"理解 Web 开发的本质"（路由、JSON、HTTP），而不是"用最流行的框架"。
>
> **为什么用 SQLite？** SQLite 是零配置的嵌入式数据库，不需要数据库服务器，适合小项目和学习项目。MySQL 和 PostgreSQL 需要额外部署和配置，对 CampusFlow 这样的项目来说是过度设计。如果未来用户量增长，可以考虑迁移到 PostgreSQL。

#### 架构设计（30 秒）

> 我们参考了 DDD（领域驱动设计），核心是"领域模型 → Repository → Controller → API"的分层架构。
>
> （展示架构图）
>
> 这样职责清晰：领域模型处理业务逻辑，Repository 负责数据访问，Controller 处理 HTTP 请求。这个架构虽然是经典的，但对学习项目很合适。

#### API 设计（30 秒）

> 我们遵循 RESTful 原则——GET 获取资源，POST 创建资源，PUT 更新资源，DELETE 删除资源。API 端点是名词（/api/tasks），不是动词（/api/getTasks）。
>
> （展示 Swagger UI 截图）
>
> 我们还用 OpenAPI 规范记录了所有端点，可以看 Swagger UI——所有端点都有文档，还能直接测试。

### 5. 踩坑与成长（1 分钟）

#### Bug Bash（30 秒）

> Week 12 跨组测试时，其他组发现了 3 个 bug：
> 1. 任务标题超过 100 字符时截断（已修复：改用省略号）
> 2. 删除任务时没有确认提示（已添加：确认对话框）
> 3. 移动端适配问题（已优化：响应式布局）
>
> Bug Bash 让我们意识到"用户的操作和我们想象的不一样"，需要更多边界测试。

#### AI 协作（30 秒）

> Week 10 我们用 AI 生成前端代码，发现了 2 个问题：
> 1. **AI 没有处理空输入**（加了边界检查）
> 2. **AI 生成的 CSS 不够美观**（人工优化）
>
> 这让我们意识到：**AI 是倍增器，但质量把控在人**。AI 能生成代码，但边界情况、安全性、可维护性——这些需要人工审查。

## 常见问题回答（备用）

### Q：为什么用 Java？不用 Python 或 Go？

> 我们团队有 Java 基础，而且 Java 的静态类型系统更适合学习工程化开发。Python 快速原型好，但大型项目维护成本高；Go 性能好，但生态不如 Java。对于学习项目，Java 是平衡选择。

### Q：为什么不用 Spring？

> Spring 是企业级框架，功能强大但学习曲线陡峭。CampusFlow 的目标是"代码简洁易懂"，Javalin 只有几百 KB，但足够 Web 开发。我们更关注"理解 Web 开发的本质"（路由、JSON、HTTP），而不是"用最流行的框架"。

### Q：你们的架构有什么特色？

> 我们参考了 DDD（领域驱动设计），核心是"领域模型 → Repository → Controller → API"的分层架构。这样职责清晰：领域模型处理业务逻辑，Repository 负责数据访问，Controller 处理 HTTP 请求。这个架构虽然是经典的，但对学习项目很合适。

### Q：CampusFlow 和 Notion / Todoist 比有什么优势？

> CampusFlow 不是"替代 Notion"，而是"学习项目"。我们的优势是：
> 1. **代码简洁**：2000 行代码 vs Notion 的百万行，容易学习和扩展
> 2. **技术透明**：所有技术选型、架构决策都有 ADR 记录，适合学习
> 3. **工程化**：测试、文档、部署流程完整，可以作为"小而美"的工程范例
>
> CampusFlow 的目标是"让学习者理解软件工程"，不是"成为商业产品"。

### Q：下一步有什么计划？

> 短期计划：增加任务搜索功能、优化移动端体验、增加数据导出。长期计划：如果用户量增长，考虑迁移到 PostgreSQL + Redis；如果有多人协作需求，增加用户认证。但我们会保持"小而美"的定位，不做功能堆砌。
