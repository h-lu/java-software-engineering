### 2. `15_showcase_poster.html`
**用途**：CampusFlow 项目集市海报（HTML 版）
**内容**：
- 可打印为 A1 尺寸的海报模板
- 远距离可读（3 米外能看清标题）
- 包含项目介绍、技术栈、亮点、二维码

**使用方式**：
1. 在浏览器中打开此 HTML 文件
2. 根据实际情况修改内容（项目名称、团队信息、二维码链接）
3. 导出为 PDF（打印 → 另存为 PDF）
4. 打印为 A1 尺寸（594 mm × 841 mm）

**设计原则**：
- 字体大小：标题 ≥ 72pt，正文 ≥ 24pt
- 配色方案：不超过 3 种主色
- 信息层级：标题 → 核心内容 → 二维码
- 视觉元素：架构图、截图、图标

**二维码生成**：
- GitHub：https://github.com/your-team/campusflow
- 在线演示：https://campusflow.up.railway.app
- API 文档：https://campusflow.up.railway.app/swagger-ui

可以使用以下工具生成二维码：
- https://www.qrcode-generator.com/
- https://api.qrserver.com/v1/create-qr-code/

### 3. `15_feedback_form.md`
**用途**：展示反馈收集表
**内容**：
- 完整的反馈评估表（演示质量、内容质量、材料质量、问答质量）
- 开放反馈部分（最印象深刻的点、最需要改进的点）
- 反馈汇总表和改进计划模板

**使用方式**：
- 打印纸质版，在展示现场发放
- 转为在线表单（Google Forms、金数据、问卷星）
- 团队内部复盘时使用

**反馈维度**：
1. **演示质量**：叙事清晰度、时间控制、表达流畅度、眼神交流
2. **内容质量**：问题背景、技术亮点、功能演示、踩坑与成长
3. **材料质量**：PPT 清晰度、海报吸引力、演示环境稳定性、二维码可访问性
4. **问答质量**：回答准确性、回答清晰度、态度友好度

**反馈使用指南**：
- 统计平均分，找出得分最低的三项
- 分类汇总：正面反馈、改进建议、问题
- 根据反馈调整：删减内容、调整语速、准备本地备份

## PPT 设计原则

### 一页一信息

- 每页只讲一个核心点
- 不要堆砌多个功能、多个概念
- 如果内容太多，拆成两页

### 文字精简

- 用关键词，不要用完整句子
- 字体大小：标题 ≥ 32pt，正文 ≥ 24pt
- 一页文字不超过 6 行

### 图胜于文

- 能用图就不用表，能用表就不用字
- 架构图、流程图、截图比文字更直观
- 用颜色高亮关键信息（但不要超过 3 种颜色）

### 一致性

- 配色方案统一（不超过 3 种主色）
- 字体风格统一（标题、正文统一用相同的字体）
- 布局风格统一（所有标题在同一位置）

## 常见问题回答

### Q：为什么用 Java？不用 Python 或 Go？

> 我们团队有 Java 基础，而且 Java 的静态类型系统更适合学习工程化开发。Python 快速原型好，但大型项目维护成本高；Go 性能好，但生态不如 Java。对于学习项目，Java 是平衡选择。

### Q：为什么不用 Spring？

> Spring 是企业级框架，功能强大但学习曲线陡峭。CampusFlow 的目标是"代码简洁易懂"，Javalin 只有几百 KB，但足够 Web 开发。我们更关注"理解 Web 开发的本质"（路由、JSON、HTTP），而不是"用最流行的框架"。

### Q：CampusFlow 和 Notion / Todoist 比有什么优势？

> CampusFlow 不是"替代 Notion"，而是"学习项目"。我们的优势是：
> 1. **代码简洁**：2000 行代码 vs Notion 的百万行，容易学习和扩展
> 2. **技术透明**：所有技术选型、架构决策都有 ADR 记录，适合学习
> 3. **工程化**：测试、文档、部署流程完整，可以作为"小而美"的工程范例

## 反馈驱动改进

### 第 1 次演练反馈

- **问题**：语速太快，内容太多，超时 2 分钟
- **改进**：删减 30% 内容，只保留核心亮点

### 第 2 次演练反馈

- **问题**：眼神交流少，一直看 PPT
- **改进**：从"背词"改成"记结构"，多练习看观众

### 第 3 次演练反馈

- **问题**：功能演示有点快，观众没看清楚
- **改进**：功能演示放慢速度，鼠标移动慢一点

### 第 4 次演练反馈

- **问题**：被问到"为什么不用 Spring"时，回答不够有说服力
- **改进**：准备 10-15 个常见问题的标准答案

### 最终展示

- 经过 4 轮反馈和改进，展示质量显著提升
- 时间控制良好（8-9 分钟）
- 叙事清晰，技术亮点突出
- 功能演示流畅，问答回答准确

## 总结

Week 15 的重点是**让别人看见你的作品**。这不是"堆砌功能"，而是"讲故事"。

通过这三个示例文件，你将学会：
1. 如何设计流畅的演示脚本（`15_presentation_script.md`）
2. 如何准备清晰的展示材料（`15_showcase_poster.html`）
3. 如何收集有价值的反馈（`15_feedback_form.md`）

记住，**代码写得再好，如果没人能用，就没有价值**。展示是让你的作品被看见、被理解、被认可的方式。

展示不是终点，而是起点。通过展示，你能获得反馈，继续改进，让作品越来越好。
