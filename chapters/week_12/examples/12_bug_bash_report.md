## 优先级定义

| 级别 | 定义 | 示例 | 修复时间 |
|------|------|------|---------|
| **P0** | 崩溃/安全漏洞 | 系统崩溃、数据泄露、SQL 注入 | 立即 |
| **P1** | 核心功能缺陷 | 无法创建任务、保存失败 | 本周 |
| **P2** | 体验问题 | UI 卡顿、提示不友好 | 下周 |
| **P3** | 优化建议 | 可以更好但不影响使用 | 技术债 |

## 附录：Bug Bash 流程

```
1. 准备（15 分钟）
   - 每个团队准备好演示环境（部署在测试服务器）
   - 准备一个"测试指南"（核心功能列表、已知问题）

2. 交叉测试（60 分钟）
   - A 团队测试 B 团队的系统，B 测试 C，C 测试 A
   - 测试者尝试"破坏"系统：输入奇怪数据、快速点击、并发操作
   - 记录每个 bug 的复现步骤

3. 汇总（30 分钟）
   - 所有团队回到一起，报告发现的问题
   - 按优先级分类（P0 崩溃 / P1 功能缺陷 / P2 体验问题）
   - 评选"最奇葩 bug"奖

4. 修复（课后）
   - 每个团队修复自己系统的 bug
   - 下周分享修复经验
```

## 附录：测试技巧

### 如何快速找到 bug？

1. **边界值测试**
   - 输入空值、null、0、-1
   - 输入超长字符串（1000+ 字符）
   - 输入特殊字符（emoji、符号、脚本）

2. **并发测试**
   - 快速连续点击按钮
   - 同时打开多个标签页操作同一数据
   - 模拟网络延迟

3. **断网测试**
   - 请求过程中断网
   - 服务器返回错误

4. **跨浏览器测试**
   - Chrome、Firefox、Safari、Edge
   - 移动端浏览器

5. **权限测试**
   - 未登录访问需要权限的页面
   - 尝试操作其他用户的数据

6. **输入验证测试**
   - SQL 注入尝试
   - XSS 脚本注入
   - 路径遍历攻击

## 附录：根因分析（Root Cause Analysis）

### 什么是根因分析？
不只是修复表面问题，而是深入分析"为什么会出现这个 bug"。

### 根因分类
| 根因类型 | 描述 | 示例 |
|---------|------|------|
| **设计缺陷** | 架构或设计导致的问题 | 没有考虑并发、没有事务隔离 |
| **代码质量** | 实现细节的问题 | null 检查缺失、边界条件处理不当 |
| **测试盲区** | 测试未覆盖的场景 | 集成测试缺失、边界情况未测试 |
| **配置问题** | 环境或配置错误 | 数据库连接字符串错误、日志级别过高 |
| **依赖问题** | 第三方库或外部服务 | API 变更、版本兼容性 |

### 根因分析工具
1. **5 Whys**：连续问 5 次"为什么"，找到根本原因
2. **鱼骨图**：从人、机、料、法、环多角度分析
3. **时序图**：分析问题发生时的调用链
