# AI 辅助编程的融合路径

## 核心理念：AI 是倍增器

本课程的 AI 理念：

> **"AI 是倍增器，不是替代者。好的工程师用 AI 放大自己的能力，但架构决策、质量把控、设计思维必须掌握在自己手中。"**

这与 Linus Torvalds 的观点一致：
- 编译器才是 1000 倍提升，AI 只是再加 10-100 倍
- AI 是工具，不是革命
- AI 对维护代码、找 bug 最有价值

也与 Grady Booch 的观点一致：
- 架构决策必须由人做
- 概念完整性需要人来维护
- AI 可以辅助，但不能替代工程判断

---

## AI 能做什么 vs 人必须做什么

| AI 擅长（倍增） | 人的责任（不可 delegating）|
|----------------|---------------------------|
| 生成样板代码 | 架构决策、ADR |
| 生成测试用例骨架 | 测试策略设计、边界情况思考 |
| 代码审查辅助（找 bug）| 质量把控、最终判断 |
| 文档生成 | 知识传递、设计意图表达 |
| 自动化重构 | 重构决策、技术债管理 |

## 全程使用策略

### 核心原则

1. **AI 使用透明化**
   - 每份作业必须包含 AI 使用日志
   - 记录：Prompt → AI 输出 → 人工审查 → 最终代码

2. **审查是必修课**
   - 不只是用 AI，要会批判性审查 AI 输出
   - Week 10 专门训练 AI 审查能力

3. **架构决策在人**
   - ADR 必须由人写，不能用 AI 代劳
   - 首席架构师做设计决策

4. **最终代码必须能解释**
   - 学生必须能解释每行代码的作用
   - 不能是"AI 写的，我不知道"

## 每周 AI 融合重点

### Week 01-02：AI 作为学习工具
- 用 AI 解释 Java 报错信息
- 用 AI 对比 Python vs Java 差异
- **但**：ADR-001 必须自己写

### Week 03-04：AI 作为编码辅助
- 用 AI 生成 getter/setter 等样板代码
- 审查 AI 生成的异常处理代码
- **重点**：识别 AI 遗漏的边界情况

### Week 05-06：AI 作为测试辅助
- 用 AI 生成测试用例骨架
- 审查：AI 生成的测试覆盖边界了吗？
- 补充 AI 遗漏的测试

### Week 07-08：AI 作为重构辅助
- 用 AI 建议重构方案
- 评估：这个重构建议合理吗？
- 决策：是否采纳（ADR 记录）

### Week 09-10：AI 审查训练（核心）⭐

**Week 10 是课程核心周**：

**上午**：AI 辅助开发
- Prompt 工程
- 用 AI 生成前端代码

**下午**：AI 审查训练（重点）
- 给一段 AI 生成的代码
- 使用审查检查清单评估
- 找出至少 3 个问题：
  - 功能问题
  - 边界情况（AI 常遗漏）
  - 安全问题
  - 设计缺陷
- 修复并改进

**作业**：
```markdown
## AI 审查训练报告

### 原始 Prompt
...

### AI 生成代码
...

### 审查发现的问题
1. 边界情况：AI 没有处理空输入
2. 安全问题：有 SQL 注入风险
3. 设计缺陷：耦合度过高

### 修复后的代码
...

### 经验总结
- AI 擅长：...
- AI 不擅长：...
- 我的判断：...
```

### Week 11-12：AI 作为质量辅助
- AI 辅助代码审查（Linus 的实验）
- AI 找 bug vs 人类找 bug 对比
- 静态分析 + AI 辅助

### Week 13-16：AI 作为项目管理辅助
- AI 辅助文档生成（需审查）
- AI 辅助 Bug 分类
- 但：架构决策、发布决策必须人来定

## AI 审查检查清单

学生必须用此清单审查所有 AI 生成代码：

```markdown
## AI 代码审查清单

### 功能正确性
- [ ] 代码是否实现了需求？
- [ ] 是否有明显的逻辑错误？

### 边界情况（AI 常遗漏）⭐
- [ ] 空输入处理了吗？
- [ ] 极值情况（最大值、最小值）处理了吗？
- [ ] 异常情况处理了吗？

### 错误处理
- [ ] 可能的异常被捕获了吗？
- [ ] 错误信息清晰吗？

### 安全考虑 ⭐
- [ ] 有 SQL 注入风险吗？
- [ ] 有 XSS 风险吗？
- [ ] 敏感信息泄露了吗？

### 可维护性
- [ ] 命名清晰吗？
- [ ] 有必要的注释吗？
- [ ] 代码重复了吗？

### AI 特定问题 ⭐
- [ ] 有"幻觉"（不存在的方法/类）吗？
- [ ] 有过时的 API 使用吗？
- [ ] 有不符合项目规范的地方吗？
```

## 与 Python 教材的差异

| Python 教材（大一） | Java 教材（进阶） |
|---------------------|-------------------|
| AI 观察 → 识别 → 协作 → 主导 | **AI 倍增器理念全程贯穿** |
| AI 练习是"可选" | AI 使用是"必需+透明+审查" |
| 重点是"学会编程" | 重点是"**学会工程化地使用 AI**" |
| 无 ADR | **ADR 必须由人写** |
| 无首席架构师 | **首席架构师轮换** |

## 关键认知

### Linus Torvalds 的观点
- "LLMs 将帮助我们写出更好的软件，更快"
- "小团队能有机会竞争大公司"
- **但**："我绝不想称 AI 为革命，编译器才是革命"

### Grady Booch 的观点
- AI 是"随机鹦鹉"，没有真正推理
- **但**：AI 在找 bug 方面很有价值
- 架构决策必须由人做

### 本课程的立场

**乐观但谨慎**：
- ✅ 积极使用 AI 作为倍增器
- ✅ 拥抱 AI  democratizing code 的趋势
- ⚠️ 但必须保持独立判断
- ⚠️ 架构决策、质量把控不能 delegating 给 AI

**最终目标**：
培养能在 AI 时代做出**正确工程决策**的工程师，而不是只会用 AI 生成代码的"提示词工程师"。
