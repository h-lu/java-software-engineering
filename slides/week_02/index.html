🚨 一个"什么都能做"的类

小北：我把创建、验证、保存、显示、邮件通知都放进了 Task 类，这不挺好的吗？

public class Task {
    public String title;
    public boolean isValid() { /* 验证 */ }
    public void saveToFile(String f) { /* 存储 */ }
    public void print() { /* 显示 */ }
    public void sendEmail(String e) { /* 通知 */ }
    // ... 还有 20 多个方法
}

老潘：你现在觉得方便，三个月后这个类会有 500 行。想加新功能都不知道插在哪。

📊 上帝类 vs 好的类设计

特征 好的类设计 ✅ 上帝类 ❌
职责数量 1-2 个相关职责 5+ 个不相关职责
方法数量 5-10 个 20+ 个
代码行数 100-200 行 500+ 行
可测试性 容易（职责清晰） 困难（依赖太多）
可维护性 高（改一处只影响一处） 低（牵一发而动全身）

⚖️ 单一职责原则（SRP）

Single Responsibility Principle
一个类应该只有一个引起它变化的原因。

阿码：职责多了有什么坏处？程序能跑啊。

老潘：想改优先级存储方式？上帝类要改 5 个地方。职责分离只需改 3 个类。

改优先级 → Task类(5处) → 测试20个方法
VS
改优先级 → Task/TaskValidator/TaskRepo → 测试3个类

📦 实体类 vs 服务类

📦

实体类（Entity）

存储数据，有状态

  • 代表问题域中的"事物"
  • 例如：Task、User
  • 特征：有字段、getter/setter
private String title;
public String getTitle()
⚙️

服务类（Service）

提供行为，无状态

  • 代表"动作"或"管理器"
  • 例如：TaskManager
  • 特征：主要是方法，字段很少
public void addTask(Task t)
public List<Task> filter()

🔓 小北的三次试错：第一次

小北：不用 private，全部 public，写起来快！

阿码：全部 public 不是更灵活吗？为什么要用 private 把自己限制住？

老潘：先看小北踩的坑，你就明白了。

❌ 全部 public

public class Task {
    public String title;
    public int priority; // 1=高, 2=中, 3=低
}
Task task = new Task();
task.priority = 5; // 手滑写成5，不报错！
// Bug藏了三天...

💥 后果

  • priority = 5 超出预期范围(1-3)
  • 程序正常运行，无报错
  • 三天后发现，影响十几个任务

老潘：public 字段没有任何保护，外部可以赋任何值。

🔓 第二次尝试：加验证，但用 public

❌ 还是可以绕过

public class Task {
    public String title;
    public int priority;
    
    public void setPrioritySafe(int p) {
        if (p < 1 || p > 3) {
            throw new IllegalArgumentException();
        }
        this.priority = p;
    }
}

🚨 问题

// 同事阿码不知道有setPrioritySafe
task.priority = 2; // 直接赋值！
// 检查形同虚设

老潘：只要字段是 public，就无法强制所有人走验证逻辑。

🔒 第三次：正确的封装

老潘：把字段藏起来，只暴露你控制的方法。

public class Task {
    private String title;      // private：外部无法直接访问
    private int priority;
    
    // 外部只能通过这个方法修改
    public void setPriority(int priority) {
        if (priority < 1 || priority > 3) {
            throw new IllegalArgumentException("优先级必须是 1-3");
        }
        this.priority = priority;
    }
    
    public int getPriority() {
        return priority;
    }
}

task.priority = 5 编译都过不了——外部只能通过 setPriority(5)，会立即抛出异常。

🎭 封装的真正价值

老潘：封装让你可以"撒谎"——内部偷偷改变实现，外部完全不知道。

初始设计

private boolean completed;

三个月后（内部变了）

private enum Status { ... }
private Status status;

老潘：但 isCompleted() 方法签名不变，外部代码完全不用改！

📐 SOLID 原则入门

老潘：我写过 1200 行的 TaskProcessor 类。加短信通知功能时，测试了 12 个功能，3 个被影响坏了。

原则 名称 一句话记住
S Single Responsibility
单一职责		 一个类只做一件事
O Open/Closed
开闭原则		 加功能不改旧代码
L Liskov Substitution 子类能替换父类
I Interface Segregation 接口小而专一
D Dependency Inversion 依赖抽象不依赖具体

本周重点：S（单一职责）和 O（开闭原则）

✂️ 拆分小北的上帝类

❌ 拆分前（上帝类）

public class Task {
    // 职责1：存储
    public String title;
    // 职责2：验证
    public boolean isValid() { }
    // 职责3：存储
    public void saveToFile(String f) { }
    // 职责4：显示
    public void print() { }
    // 职责5：通知
    public void sendEmail(String e) { }
}

✅ 拆分后（5个类）

Task - 存储数据
TaskValidator - 验证
TaskRepository - 持久化
TaskPrinter - 显示
EmailNotifier - 通知

老潘：现在每个类只有一个职责。改验证规则只需动 TaskValidator。

🔓 开闭原则（OCP）：对扩展开放

❌ 不符合 OCP

public class TaskPrinter {
    public void print(Task t, String format) {
        if (format.equals("text")) {
            // 文本格式
        } else if (format.equals("json")) {
            // JSON格式
        }
        // 加新格式要修改这个类
    }
}

✅ 符合 OCP

interface TaskFormatter {
    String format(Task task);
}

class TextFormatter implements TaskFormatter { }
class JsonFormatter implements TaskFormatter { }

// 加XML格式？新建类，不用改TaskPrinter
class XmlFormatter implements TaskFormatter { }
核心思想：通过某种方式扩展功能，而不是修改现有代码

🤖 AI 时代小专栏 #2

SOLID 原则在 AI 时代仍然重要

⚠️ AI 代码的问题

  • 平均节省 55% 时间，但质量下降
  • 相当比例存在"职责不清晰"
  • 代码重复量显著增加

✅ SOLID 作为审查标尺

  • S：这个类是不是承担太多职责？
  • O：需求变了需要修改吗？
  • 用来判断"AI 代码是否值得接受"

SOLID 不是过时的理论，而是 AI 时代的核心技能。

📋 ADR：记录"为什么"

小北：为什么要写文档？代码不就是最好的文档吗？

老潘的故事

看到 validateAndSaveAndNotify()，我嫌太长就拆开了。

结果系统崩了——原来三者必须在同一事务里执行。

但代码里没有任何说明，设计者也离职了。

老潘：代码说"怎么做"，ADR 说"为什么"。

📝 ADR 的基本结构

状态 已采纳 / 已废弃 / 已替代
背景 要解决什么问题
决策 我们决定怎么做
理由 为什么这样做
替代方案 考虑过什么，为什么不选
后果 正面、负面、风险
验证方式 如何验证决策正确

重点写"为什么选 A 而不是 B"——这是最有价值的信息

❌ 被老潘打回的版本

# ADR-001: TaskManager 设计

## 决策
设计了 Task 类和 TaskManager 类。

## 理由
这样设计比较好，符合 SOLID 原则。

## 替代方案
无。

"比较好"是什么意思？哪个原则？考虑过其他方案吗？

✅ 修改后的版本

  • ☑️ 具体解决了什么问题
  • ☑️ 哪个原则，如何符合
  • ☑️ 考虑过 public 字段、Lombok
  • ☑️ 正面/负面/风险
  • ☑️ 验证方式（Week 04/07）

💻 Task.java 代码示例

package edu.campusflow.domain;

public class Task {
    private final String title;
    private final String description;
    private boolean completed;

    public Task(String title, String description) {
        this.title = title;
        this.description = description;
        this.completed = false;
    }

    public String getTitle() { return title; }
    public String getDescription() { return description; }
    public boolean isCompleted() { return completed; }
    
    public void markCompleted() {
        this.completed = true;
    }
}
✅ private 字段 ✅ final 不可变 ✅ 语义化方法名
